# TERMS

This document defines common terms used throughout the **aieos-release-exposure-kit** repository.
These definitions are intentionally **tool-agnostic**, **employer-neutral**, and **AI-friendly**.

---

## Core Concepts

### Release
The controlled introduction of a production-ready system, feature, or change to production users. A release is not the same as a deployment — deployment places code in a production environment; release exposes it to users.

---

### Deployment
The technical act of placing new software artifacts, configuration, or infrastructure changes into a production environment. Deployment is a prerequisite for release, not equivalent to it.

---

### Exposure
The percentage of production users or traffic that can access a newly deployed feature or service. Exposure starts at 0% after deployment and increases through defined stages.

---

### Progressive Delivery
A release strategy in which exposure is increased incrementally, with observation periods between stages, allowing problems to be detected at small scale before reaching full exposure.

---

### Rollback
The act of reversing a deployment or reducing exposure to zero in response to a rollback condition being triggered. Rollback restores the system to its previous stable state.

---

### Rollback Condition
A concrete, observable signal with a defined threshold that, when met, requires a rollback decision to be made. Rollback conditions are defined in the Release Plan before execution begins. Qualitative assessments ("if things look wrong") are not rollback conditions.

---

## Artifact Types

### RER — Release Entry Record
A lightweight gate completed before release planning begins. Confirms the upstream ORD is frozen, declares the release scope and type, names the release owner, and confirms an authorization decision exists on record.

The RER is human-authored. It is not generated by AI.

---

### RCF — Release Context File
An organizational policy artifact defining the deployment standards, exposure standards, authorization model, communication requirements, and monitoring requirements that govern all releases within its stated scope.

The RCF is reusable across multiple releases. It is authored at the organizational level and updated only when release policy changes.

---

### RP — Release Plan
A per-release artifact defining the deployment strategy, exposure stages, rollback conditions, communication plan, and monitoring protocol for a specific release engagement.

The RP is generated from a frozen ORD and frozen RCF. A frozen RP, combined with a completed Pre-Release Authorization Checklist, authorizes execution to begin.

---

### RR — Release Record
A post-release evidence artifact documenting what actually happened during a release. Captures deployment evidence, exposure decisions, monitoring observations, and final disposition.

The RR is an evidence artifact — it does not revise the Release Plan. Deviations from the RP are documented in the RR; the RP remains frozen.

---

### ORD — Operational Readiness Document
The terminal artifact of the Engineering Execution Kit (Layer 4). Confirms the system is production-ready: deployment procedures tested, observability in place, rollback procedures verified. The ORD is the upstream input to the Release & Exposure Kit.

---

## Release Concepts

### Pre-Release Authorization Checklist
A section of the Release Plan completed by the release owner immediately before execution begins — not at plan freeze time. It is the go/no-go gate that authorizes execution.

---

### Exposure Stage
A defined increment in the progressive delivery rollout. Each stage has a target exposure percentage (or audience segment), a proceed trigger (observable signal with threshold), and a hold action if the trigger is not met.

---

### Proceed Trigger
The observable signal and threshold that must be met before exposure is increased to the next stage. Proceed triggers are defined in the RP and evaluated against monitoring data during the observation period.

---

### Observation Period
The minimum time that must elapse, with active monitoring, before a stage is closed and exposure is increased. Defined in the RCF as a clock duration and referenced in the RP per stage.

---

### Watch Period
The minimum time the release team must actively monitor the system after each exposure stage. Equivalent to the observation period requirement from the RCF. Used in the Release Plan (§6) and reported in the Release Record (§5).

---

### Feature Flag
A runtime configuration mechanism used to control which users are exposed to a feature. Feature flags enable progressive delivery by decoupling deployment (code is in production) from exposure (users can access the feature).

---

### Flag Lifecycle
The defined progression of a feature flag: inactive → percentage rollout → segment targeting → full exposure → removed. Flags must be removed within the RCF-specified period after reaching full exposure.

---

### Release Disposition
The formal declaration of a release outcome, recorded in the Release Record. One of: successful-full-exposure | successful-partial-exposure | rolled-back | abandoned.

---

### Handoff to Layer 6
The section of the Release Record (§7) that provides the Reliability & Resilience Kit with everything it needs to begin production oversight: what is in production, current exposure state, active monitoring, SLO baseline, open incidents, and recommended watch items.

---

## Evidence Standards

### Evidence (Release Record)
An artifact entry that is: concrete (not an assertion), timestamped (date and time), traceable (references the RP step or condition it documents), and retrievable (includes a location where the artifact can be accessed).

---

### Assertion
A claim without supporting artifact. Assertions are not acceptable as evidence in the Release Record. "Deployment completed successfully" is an assertion. "kubectl rollout status output at 14:07 UTC — stored at releases/rr-001/deploy.log" is evidence.

---

## Validation

### Hard Gate
A specific, binary pass/fail criterion in a spec. All hard gates must pass for a validator to return PASS status. A single failing gate fails the artifact.

---

### Standard Enforceability
A property of a Release Context File: all standards in the RCF must be specific enough to evaluate compliance without judgment. Standards using "should," "ideally," "when possible," or equivalent aspirational language fail this criterion.

---

## Key Principles

- **AI generates, validators decide**
- **Rollback conditions are observable signals, not judgments**
- **Evidence is concrete, timestamped, traceable, and retrievable**
- **Freeze before promote — upstream artifacts must be frozen before downstream generation**
- **Deviations are documented in the RR; the RP remains frozen**
- **Separate generation and validation sessions to prevent self-validation bias**

---

## Notes

- These terms are intentionally generic and tool-agnostic
- No terminology implies a specific employer, vendor, or observability platform
- Examples in this repository use placeholder names and neutral contexts
