## 3. Exposure Specification

**Exposure strategy:** percentage rollout via feature flag NOTIF_ENABLED

| Stage | Target | Trigger to Proceed | Hold Action if Trigger Not Met |
|-------|--------|-------------------|-------------------------------|
| Stage 1 | 10% of users (random sample) | Error rate <0.1% AND p99 latency <300ms, both sustained for 4 hours | Investigate with SRE before proceeding; do not expand to Stage 2 |
| Stage 2 | 25% of users (random sample) | Error rate <0.1% AND p99 latency <300ms AND notification delivery rate >98%, all sustained for 4 hours | Pause expansion; investigate and resolve before proceeding |
| Stage 3 | 100% of users (full exposure) | 4-hour watch period complete at 25% with all Stage 2 success criteria met | Investigate; escalate to Engineering Manager if not resolved within 2 hours |

**Feature Flag Configuration:**

| Flag Name | Stage 1 Value | Stage 2 Value | Stage 3 Value | Set By |
|-----------|--------------|--------------|--------------|--------|
| NOTIF_ENABLED | 10% | 25% | 100% | Sarah Chen via LaunchDarkly |
| NOTIF_BATCH_SIZE | 50 | 50 | 100 | Sarah Chen via LaunchDarkly (increase at Stage 3) |

**RCF Compliance:**
- Maximum increment: Stage 1 = 10%, Stage 2 = +15% (25% total), Stage 3 = +75% (100% total) — Stage 3 exceeds 25-point maximum
- **Stage 3 exception justification:** Direct jump from 25% to 100% is a 75-point increment, which exceeds the RCF-TASKFLOW-001 §3 maximum of 25 percentage points. This exception is requested per RCF §3 Tier 2 exception criteria: the notification feature has been at full exposure in staging for 14 days with zero incidents. Exception authorized by David Park (Engineering Manager) on 2024-03-15 as part of release authorization (reference: SPR-2024-Q1-R03).
- Observation period: 4 hours per stage ≥ RCF minimum of 4 hours ✓

## 4. Rollback Specification

**Rollback Conditions:**

Rollback must be triggered when any of the following conditions is observed:

| Condition | Signal | Threshold | Sustained Duration |
|-----------|--------|-----------|-------------------|
| 1 | Application error rate (5xx responses) | >0.5% | 5 minutes sustained |
| 2 | p99 API latency (notification endpoints) | >500ms | Any single measurement at or above threshold during a 15-minute check-in |
| 3 | Notification delivery rate | <95% | 15 minutes sustained |
| 4 | JavaScript error rate (browser console errors from notification UI) | >1% of page loads | 10 minutes sustained |

**Rollback Procedure:**

Reference: ORD-TASKFLOW-001 §8 Rollback procedure, steps 1–5

Summary:
1. Set NOTIF_ENABLED to 0% in LaunchDarkly (immediate — takes effect within 60 seconds for all users)
2. Scale canary notification-service instances to 0 (`kubectl scale deployment notification-service-canary --replicas=0`)
3. Verify: error rate returns to pre-release baseline within 10 minutes; p99 latency returns to baseline
4. Set NOTIF_BATCH_SIZE to inactive in LaunchDarkly
5. Confirm rollback complete: all traffic returning through notification-service v1.3.2 (non-canary instances)

**Responsible Party:** Sarah Chen — backup: Marcus Rivera (SRE on-call)

**Decision Authority:** Sarah Chen authorizes rollback decisions at any exposure level. If Sarah is unavailable, Marcus Rivera (SRE on-call) has rollback authority. Escalation: David Park (Engineering Manager) if neither is reachable.

**Decision Window:** Rollback decision must be made within 15 minutes of a condition being triggered. If no decision is made within 15 minutes, Marcus Rivera initiates rollback automatically.
